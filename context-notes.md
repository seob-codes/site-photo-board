## 1. 프로젝트 개요

- **프로그램명.** 현장보드판 (fieldboard)
- **목적.** 건설/유지보수 현장에서 사진을 찍으면서 작업 정보(접수번호·단지명·동/호·공종·위치·하자내용·촬영구분)를 입력해 규칙에 맞는 파일명으로 구글 드라이브에 자동 업로드하는 모바일용 웹앱. 사진 위에는 실제 현장 보드판처럼 표(단지명/동-호/위치/작업사항/비고)가 합성됨.
- **사용자.** 나 + 현장 작업자 — 외부 배포(GitHub Pages)로 운영 중이므로 깨지면 바로 현장 업무에 영향.
- **주요 파일.**
  - `fieldboard-v4.html` — 전체 앱(HTML+CSS+JS 단일 파일). 카메라 촬영, 표 오버레이 합성, 항목별 음성 입력, 목록 관리, 작업목록, 드라이브 업로드, 설정 화면, 완료확인서 모두 포함. **기능 수정은 거의 항상 이 파일만 다시 업로드하면 됨.**
  - `manifest.json` — PWA(홈 화면 앱) 설정 파일. 앱 이름/아이콘/색상/실행방식 정의. 거의 안 바뀜.
  - `sw.js` — 서비스워커. 네트워크 우선 캐싱(인터넷 되면 항상 최신 파일 우선) 담당. 거의 안 바뀜.
  - `icon-192.png`, `icon-512.png` — PWA 아이콘(카메라 모양, 기본형). 거의 안 바뀜.

## 2. 현재 상태 (2026-06-25 기준)

- **완료확인서 양식 좌표 미세조정 다수 완료**: `drawCertificate()`의 표 7줄 높이를 50→70→60으로 시행착오 끝에 60으로 확정(접수내용/보수내용은 130로 확대), 제목 시작 y좌표를 90→120→150으로 점점 내려서 위쪽 여백 확보. 확인문구/날짜/확인자 줄 간격도 여러 차례 조정 끝에 확정(작업자/현장대리인 값을 칸 가운데 정렬로 변경 포함). **모든 좌표 변경은 `node -e`로 직접 시뮬레이션 계산해서 검증** — 과거엔 계산 실수로 안내 그림 숫자가 50px씩 틀렸던 적이 있어서(써비가 카톡 스크린샷으로 직접 발견), 이후로는 추측 대신 항상 코드 실행으로 확인하는 절차로 바꿈.
- **현장대리인 도장 모양/위치 버그 수정**: 사각 테두리(`strokeRect`)였던 걸 세로형 타원(`ellipse`)으로 교체. 도장 위치가 고정 좌표라 이름 길이에 따라 "(인)" 글자와 안 겹치던 문제를 `ctx.measureText()`로 실제 글자 폭을 측정해 동적으로 계산하도록 수정. 확인자(입주자)/(관리소) 줄도 "(인)" 글자의 우측 끝이 표 우측 테두리선에 정확히 맞도록 `textAlign='right'`로 변경.
- **"공사 위치 마커 안내 그림"(`app-ui-guide-5-cert-modal.svg`, `app-ui-guide-6-cert-form.svg`) 신규 생성**: 촬영 탭의 완료확인서 미리보기 화면과, 양식 자체(표/줄 구성)를 좌표값까지 표시해서 보여주는 안내 그림. 좌표를 시각적으로 보면서 "이 줄을 얼마나 내릴지" 가늠할 수 있게 함. **코드가 바뀔 때마다 안내 그림도 같이 최신화**(이전 좌표 값이 남아있으면 다음 대화에서 혼란을 유발하므로).
- **"기기 코드 승인" 기능을 실제로 활성화함** (이전엔 `LICENSE_CHECK_URL`이 빈 값이라 검사 자체가 꺼져있었음):
  - 허가 기기 목록을 Apps Script 코드 안 배열(`ALLOWED_DEVICES`)이 아니라 **구글 시트**(`SitePhotoBoard 허가목록`, 시트ID `1EeElUMyOZexzUoITQVe6_VvqRp13RaTTGvvSSIdbu3I`)로 관리하도록 변경 — A열(A2부터)에 기기 코드를 적으면 승인, `ALLOW_ALL`이 있으면 전체 허용. 코드 재배포 없이 시트만 수정하면 되는 구조로 바꿔서 비개발자인 써비가 직접 관리 가능해짐.
  - `LICENSE_CHECK_URL` 별도 상수를 없애고, 이미 설정 탭에 입력된 드라이브 연동 주소(`#s-url`)를 그대로 재사용하도록 변경(같은 Apps Script 하나가 사진 업로드 + 기기 승인 검사를 둘 다 처리).
  - **중대 버그 발견 및 수정**: 기기 코드 검사가 항상 통과되는 문제가 있었음 — 원인은 `fetch` 요청에 `Content-Type: application/json` 헤더를 쓰면 브라우저가 CORS 사전검사(OPTIONS)를 먼저 보내는데, Apps Script가 이걸 처리하는 코드가 없어서 405로 거부되고, 그 오류가 fail-open(네트워크 오류 시 통과) 로직에 걸려 조용히 통과됐던 것. 네트워크 탭으로 직접 확인 후 `Content-Type: text/plain;charset=utf-8`로 바꿔서 해결 — **사진/완료확인서 드라이브 업로드 코드도 같은 패턴이라 동일하게 수정함** (그동안 같은 문제로 조용히 실패했을 가능성 있음, 갤러리 저장 덕분에 체감 못 했을 수 있음).
  - 검사 끝날 때까지 화면 전체를 가리는 로딩 가림막(`#license-gate`, "🔒 사용 승인 확인 중...") 신규 추가 — 기존엔 카메라 화면이 먼저 보이고 한참 뒤에 잠금이 덮이는 구조라 승인 안 된 기기도 잠깐 화면을 볼 수 있었음, 이제는 검사 끝나기 전엔 아무것도 안 보임.
  - 잠금 화면에 라이선스 경고 박스(전화번호 포함) + 기기 코드 복사 버튼 추가.
  - **무단 사용 방지의 기술적 한계를 사용자와 논의**: 클라이언트(브라우저) 전용 웹앱 구조상, 소스를 통째로 복사해서 검사 코드를 지우거나 네트워크 요청을 차단하면 우회 가능함을 설명. `.exe`(컴파일된 네이티브 앱)는 코드가 기계어로 숨겨져서 훨씬 우회가 어렵다는 점도 설명함. 지금 구조는 "링크가 그냥 퍼지는 것"을 막는 1차 방어선이라는 현실적 목표로 합의.
  - GitHub 저장소를 Private로 바꾸는 것도 논의했으나, 무료 계정 기준 GitHub Pages 배포에 영향(배포된 웹앱이 안 열릴 위험)이 있어서 **Public 상태 유지로 결정**.

## 2-0. 이전 상태 (2026-06-24 기준, 참고용)

- **"공사 위치 마커" 기능 대대적 재설계 완료** (이전엔 "연습/디자인 확정 전" 단계였음, 이번 작업으로 거의 확정 단계로 발전):
  - 모양: 네모(⬜) 삭제, 원(⭕)/화살표(➜)/텍스트(Ｔ) 3가지만 남음. 원은 채우기 없이 얇은 빨간 선만(흰 테두리는 사용자 피드백으로 제거), 화살표는 이모지 대신 직접 그린 모양(고정 크기 화살촉 + 늘어나는 선)이라 **360도 자유 회전 + 길이조절**이 가능해짐. 색상은 주황 삭제하고 노란색 추가.
  - **마커를 여러 개 동시에 사용 가능**해짐 — 이전엔 1개만 저장(`A.tapMarker`)했는데, 배열(`A.tapMarkers`)로 바꿔서 원 마커+화살표 마커 등을 한 사진에 동시에 찍을 수 있음. 빈 곳을 탭하면 새 마커가 추가되고(기존 마커는 그대로), 마커를 탭하면 점선 테두리로 "선택"되고, "지우기" 버튼은 **선택된 마커 1개만** 삭제(`deleteActiveMarker`). 재촬영(`retake`)하면 전체 초기화(`clearAllMarkers`).
  - 마커 조작 방식: **짧게 눌러 바로 끌면** 크기/회전 조절(원·텍스트는 가로세로 따로, 화살표는 중심 기준 손가락 방향=각도·거리=길이), **0.45초 이상 가만히 누르고 있다가 끌면** 자유 이동. 모양/색상 패널에서 고르면 선택된 마커에도 바로 적용됨.
  - **움직임이 부자연스럽다는 피드백을 받고 원인 2가지를 고침**: (1) 이동 시 처음 누른 지점과 마커 중심의 차이를 기억하지 않아 마커가 손가락 위치로 "튀던" 문제 → 오프셋을 기억해서 해결. (2) 화살표 회전을 절대 각도로 계산해서 마커 중심 가까이 잡으면 손 떨림에도 각도가 휙휙 튀던 문제 → "처음 잡았을 때 각도 대비 상대 회전량"으로 계산해서 해결. 추가로 드래그 중 매번 HTML을 새로 그리던 것(`innerHTML` 재생성)을 모양이 안 바뀌면 기존 요소 스타일만 갱신하는 방식(`buildMarkerSkeleton`/`renderMarkerEl`)으로 바꿔서 더 가벼워짐.
  - 텍스트 마커 입력칸: 최대 글자수 6→10자, 입력칸 너비도 넓혀서 글자가 안 짤리게 함.
- **화면 구조를 라벨로 보여주는 "안내 그림"(SVG 이미지 파일) 신규 도입**: 써비가 기능 이름을 말로 표현하기 어려워해서, 마커 패널/헤더·탭/촬영 탭/작업목록 탭/설정 탭 구조에 이름표를 붙인 그림 5개를 프로젝트 폴더에 생성(`marker-ui-guide.svg`, `app-ui-guide-1-header-tabs.svg` ~ `app-ui-guide-4-settings-tab.svg`). **CLAUDE.md에 "UI 안내 그림 자동 갱신 규칙" 추가** — 앞으로 화면에 보이는 요소(버튼/탭/입력칸 등)가 바뀌는 수정을 하면, 요청 없이도 관련 안내 그림을 같이 다시 만들어 덮어쓰기로 합의함.
- **작업 이력이 10개를 넘어서 `archive-notes.md` 신규 생성** — 2026-06-19~20의 오래된 작업 이력 11건을 그쪽으로 옮기고, 이 파일의 "14. 작업 이력" 표는 최근 10개만 유지하는 본래 관리 규칙대로 정리함.

## 2-1. 이전 상태 (2026-06-20 기준, 참고용)

- **PWA(홈 화면 앱) 전환 완료** — `manifest.json`(앱 이름/아이콘/색상/세로고정), `sw.js`(서비스워커), `icon-192.png`/`icon-512.png` 4개 파일 신규 생성, `fieldboard-v4.html` `<head>`에 manifest 연결+서비스워커 등록 스크립트 1줄 추가. 사용자가 폰에서 기존 바로가기를 지우고 "홈 화면에 추가"를 다시 해서 **설치 성공 확인함**.
  - 서비스워커는 **네트워크 우선** 방식으로 설계 — 인터넷 되는 상태에서 앱을 껐다 켜면 항상 최신 파일을 먼저 가져와 자동 반영, 인터넷이 안 될 때만 마지막 캐시를 보여줌. 새 버전 배포 시 이전 캐시는 자동 정리(`activate` 이벤트에서 구버전 캐시 삭제).
  - **앞으로 기능을 수정할 때는 `fieldboard-v4.html` 파일 하나만 다시 업로드하면 됨** — manifest/아이콘/sw.js는 AI가 따로 "이것도 같이 올려주세요"라고 안내하지 않는 한 그대로 둬도 됨.
  - PWA는 앱스토어에서 설치하는 별도 프로그램이 아니라, 브라우저의 "홈 화면에 추가" 기능으로 만들어지는 것임(사용자가 헷갈렸던 부분, 설명 후 이해함).
- **앱 이름을 "써비의 현장보드판"으로 변경** — 브라우저 탭 제목(`<title>`)과 화면 상단 헤더 둘 다 적용.
- **완료확인서 기능 신규 추가** (사용자가 첨부한 `완료확인서양식.xlsx`를 참고해 동일 레이아웃으로 이미지 생성):
  - 설정 탭에 "완료확인서 정보" 카드 추가 — 공사명/보수업체/작업자/현장대리인 4개 입력칸(한 번 입력하면 재사용, 언제든 수정 가능)
  - 촬영 화면 셔터 옆에 "📄" 버튼 → `generateCertificate()` → Canvas로 양식과 같은 레이아웃(공사명/보수업체/접수번호/단지명(동,호)/접수내용=하자내용/보수내용=공종+위치+하자내용 조합/보수완료일=오늘날짜/작업자·현장대리인/확인문구/확인자 서명란) 생성
  - **바로 저장하지 않고 먼저 미리보기 확인창**(`#cert-modal`)을 띄움 — 이미지 미리보기 + 파일명 + "돌아가기/저장" 버튼. 촬영 확인창과 동일한 디자인 패턴.
  - 확인창에 **"✍️ 입주자 서명" / "✍️ 관리소 서명" 버튼**(서로 독립적, 순서 무관) — 누르면 이름 입력(🎤 마이크 지원) + 손가락/마우스로 그리는 서명 패드(`#sig-modal`)가 뜨고, 완료하면 확인서 이미지의 "확인자(입주자)/(관리소) (인)" 위치에 이름+서명이 **"(인)" 글자에 겹치도록** 합성됨.
  - 파일명 규칙: `접수번호_단지명_동_호_완료확인서_001` (같은 접수번호로 또 만들면 002, 003…) — `buildCertFname()`/`nextCertSeqFor()`.
  - 저장하면 사진처럼 폰 갤러리 다운로드 + IndexedDB(`fb4_photos`) 저장 + `job.certs` 배열에 기록.
- **작업목록에 "확인서" 도트/버튼 추가**: 전/중/후 도트 옆에 4번째 "확인서" 도트(후가 끝나야 ▶ 활성화, 만들면 보라색 ✓). 액션 버튼 행에도 "확인서" 버튼 추가하되, **도트=보기(있으면 미리보기, 없으면 만들러 가기) / 버튼=항상 추가로 만들러 가기**로 역할을 분리해 중복 없앰(전/중/후 버튼과 동일한 패턴).
- **작업 삭제 기능 추가**: 작업 카드에 🗑️ 버튼 → 처음엔 브라우저 기본 `confirm()`을 썼다가 "조잡하다"는 피드백으로 **촬영 확인창과 같은 디자인의 삭제 확인 팝업**(`#delete-modal`)으로 교체. 데이터는 `A.jobs`에서 제거(IndexedDB의 사진 자체는 안 지움, 영향 미미).
- **공유(내보내기) 기능 추가**: 작업 카드에 "📤" 버튼 → `exportJob()` → 그 작업의 전/중/후 사진+확인서를 전부 IndexedDB에서 불러와 `navigator.share()`(폰 기본 공유창)로 내보냄. 완전 자동 문자 발송은 브라우저 보안상 불가능해서, "공유창에서 문자/카톡 선택 → 마지막 전송만 사용자가 누르는" 방식으로 타협. 지원 안 하는 브라우저에서는 안내 토스트만 뜨고 다른 기능엔 영향 없음(불편하면 버튼+함수만 빼면 바로 제거 가능).
- 작업 카드 디자인 정리: 공종 키워드 기반 이모지 아이콘(`getJobIcon`) 삭제, 도트 사이를 잇던 초록/주황 연결선(`.progress-line`) 삭제 — 사용자가 "조잡/불필요해 보인다"고 판단.
- 작업목록 도트 라벨에서 "N장 · 시간" → **"N장"만 표시**(시간 제거).
- **접수번호/동·호 입력에 "예외 텍스트" 허용**: 접수번호는 평소엔 숫자만 입력하면 기존처럼 6-2-5 자동 대시 포맷이 적용되지만, "선조치"/"관리소요청"/"소장님요청"처럼 **한글이 섞이면 자동 포맷을 건너뛰고 그대로** 입력됨(타이핑/마이크 둘 다 적용, `formatReceipt()`/`applyVoiceField()` 수정). 마이크로 한글 예외 문구를 말하면 띄어쓰기는 제거되고 붙여서 저장됨(예: "관리소 요청" → `관리소요청`).
- **동/호 마이크도 한글 인식 가능**: 기존엔 숫자("303동 704호"/"303 704")만 인식했는데, 숫자가 없으면 말한 단어를 앞 단어=동, 뒤 단어=호로 나눠서 한글도 인식하도록 확장(예: "관리동 기타" → 동: 관리동, 호: 기타).
- **접수번호 자동 포맷 버그 발견 및 수정**: 한글 예외 기능 추가 직후, 숫자만 입력해도 대시(-)가 들어간 순간부터 그 대시까지 "예외 텍스트"로 잘못 인식되어 자동 포맷이 멈추는 버그가 있었음(사용자가 직접 발견). `formatReceipt()`의 예외 판단 조건을 `/[^0-9]/` → `/[^0-9-]/`로 수정해서 해결 — 자동 삽입된 대시는 예외 판단에서 제외.
- **설정 탭 placeholder(예시 텍스트) 전부 제거**: 조직명/공사명/보수업체/작업자/현장대리인 5개 입력칸 + 목록 관리(단지명/공종/위치/하자내용) 4개 입력칸의 "예) ..." 안내 문구 모두 삭제, 빈 칸으로 표시.
- **"파일명 설정" 카드 → "워터마크 설정"으로 이름 변경**: 실제로는 파일명이 아니라 사진 워터마크(조직명 표시) 용도라서 이름이 실제 기능과 맞지 않았음, 정확하게 수정.
- **작업 카드 제목을 "공종" → "하자내용"으로 변경**: `getOrCreateJob()`에서 `job.title`을 `meta.worktype` 대신 `meta.defect`로 설정(신규 생성/기존 갱신 둘 다). 단, 기존에 이미 만들어진 카드는 그 접수번호로 다시 촬영하기 전까지는 예전 제목이 남아있음(정상 동작).
- **"산출식" 입력칸 신규 추가**(물량 칸 위): `(2*2)*2.3-(1*1)` 같은 사칙연산+괄호 수식을 입력하면 자동 계산되어 **물량 칸에 결과가 자동으로 채워짐**(`calcFormula()`). 산출식을 지우면 물량도 같이 비워짐. 마이크도 지원 — "2 곱하기 2" 같이 말하면 기호(`*`)로 바꿔서 계산까지 자동 실행(`voiceForFormula()`, 곱하기/나누기/더하기/빼기 인식). 보드판 라벨이 "물량" → **"산출식/물량"**으로 바뀌고, 산출식이 있으면 표에 **"산출식=물량"**(예: `(2*2)*2.3-(1*1)=5.6`) 형태로 표시됨.
- **작업목록에 "진행중/완료" 필터 탭 추가**: 기존엔 전/중/후 다 끝난 작업을 무조건 숨겼는데, 실제로 "후" 찍자마자 카드가 사라지는 게 당황스럽다는 피드백을 받고 — 화면 상단에 "진행중"/"완료" 탭을 추가해서 완료된 작업도 "완료" 탭에서 모아볼 수 있게 변경(`A.jobFilter`, `setJobFilter()`). 데이터는 항상 보존, 필터만 화면에 보일 목록을 결정.
- **무단 배포/사용 방지를 위한 "기기 코드 승인" 기능 신규 추가**: 지인에게 공유한 앱이 또 다른 사람에게 퍼지는 걸 막기 위해, 각 기기마다 고유한 "기기 코드"(`getDeviceId()`, localStorage에 저장)를 만들고, 앱 시작 시 `LICENSE_CHECK_URL`(개발자가 직접 입력해야 하는 Apps Script 주소)로 그 코드가 허가됐는지 확인(`checkLicense()`). 허가 안 되면 전체 화면 잠금(`#lock-overlay`)이 뜨고 기기 코드+안내 문구만 보임, 네트워크 오류 시엔 안전하게 통과(현장에서 인터넷 문제로 먹통되는 것 방지). 설정 탭에 "기기 코드" 카드(복사 가능) 추가. **Apps Script 코드(설정 탭 "📋 코드 복사" 안)에도 `ALLOWED_DEVICES` 허가목록 + `checkAccess` 처리 로직을 추가해야 동작함** — 아직 `LICENSE_CHECK_URL`이 빈 값이라 실제로는 검사가 꺼져있는 상태(개발자가 직접 URL을 채우고, 본인 기기 코드를 허가목록에 추가 + Apps Script 재배포해야 활성화됨).
- **개발자 credit/라이선스 경고 문구 추가**: 설정 탭 맨 아래에 주황색 경고 박스로 "이 프로그램은 개인 라이선스 소프트웨어입니다. 무단 복제·배포·사용을 금지합니다. 위반 시 법적 책임을 질 수 있습니다. 개발자: 김태섭" 추가(전화번호는 공개 배포 코드에 노출되는 보안 위험 때문에 제외).
- **완료확인서 레이아웃 세부 조정** (`drawCertificate()`): 접수내용/보수내용 가운데정렬을 시도했다가 사용자가 "취소, 기존(왼쪽 정렬)으로" 되돌림. 보수내용 구성을 공종-위치-하자내용 → **위치-하자내용-물량**으로 변경(`getFieldMeta()`에 quantity 추가). "작업자" 줄 라벨 너비를 위쪽 "보수완료일"과 통일해서 세로선 일치. 현장대리인 이름 옆에 "(인)" 텍스트를 유지하고, 그 위에 겹치도록 **세로로 긴 빨간 테두리 도장**(`drawVerticalStamp()`, 이름을 한 글자씩 세로로 그림)을 합성. 확인자(입주자)/(관리소) 줄을 오른쪽으로 이동, 이름~(인) 사이 간격을 좁히고 두 줄 사이 간격도 조정.
- **촬영 화면 "공사 위치 마커" 기능 신규 추가** (연습/디자인 확정 전 단계, 사용자가 현장 사진 3장을 보내 디자인 논의):
  - 셔터 옆 "📍" 버튼으로 **마커 모드**를 켜야만 화면 탭이 마커로 인식됨(평소엔 화면을 자유롭게 탭해도 안전, `A.markerMode`/`toggleMarkerMode()`)
  - 마커 모드를 켜면 패널에서 **모양(원/화살표/네모/텍스트)**, **색상**(5가지)을 선택 가능(`setMarkerShape()`/`setMarkerColor()`)
  - 처음엔 ＋/－ 버튼으로 크기 조절했다가, 사용자가 "마커를 직접 눌러서 드래그로 크기조절, 가로/세로 따로(원→타원, 화살표 길게) 가능했으면 좋겠다"고 요청 → 마커 오른쪽 아래 **드래그 손잡이**(`marker-resize-handle`)로 교체, `A.markerStyle`/`A.tapMarker`를 `size`(숫자1개) → `width`/`height`(가로세로 따로)로 재구성
  - 마커 본체를 누르면 "이동"(카메라 화면 탭), 손잡이만 누르면 "크기조절"로 역할 분리
  - 촬영 시 사진에도 같은 위치·모양·색상·크기로 합성(`drawTapMarkerOnCanvas()`, 화면 px↔사진 해상도 비율 변환)
  - **중대 버그 발견 및 수정**: 드래그 종료 처리 함수(`_markerDragEnd`)가 드래그 중이 아닐 때도 모든 mouseup/touchend에 `preventDefault()`를 호출해서, **앱 전체의 모든 버튼/탭 클릭이 막히는 사고**가 있었음(GitHub Pages에 배포 후 발견). `if(!_markerDrag.active) return;` 조건을 추가해 드래그 중일 때만 막도록 수정해서 해결. **교훈**: 전역(document) 이벤트 리스너를 추가할 때는 반드시 "조건 없이 항상 실행되는 부분이 있는지" 직접 다시 읽고 확인할 것 — 문법 검사(node -e)만으로는 이런 로직 버그를 못 잡음.

## 2-2. 이전 상태 (2026-06-19 기준, 참고용)

- 사진정보 입력란이 **9항목**(접수번호/단지명/동/호/공종/위치/하자내용/물량/촬영구분)으로 확장. 물량은 파일명 생성에는 포함되지 않는 표시 전용 항목. 촬영구분은 "완료" 제거하고 전/중/후/점검 4개만 사용.
- 사진 오버레이를 어두운 반투명 텍스트 박스에서 **흰 배경 표** 디자인으로 전면 교체, 라벨 칸도 회색(#dcdcdc)에서 **흰색**으로 통일. 표 구성은 **단지명/동-호/위치/하자내용/작업사항/물량** 6줄(위치~작업사항 사이에 하자내용, 작업사항 다음에 물량 추가). 라이브 카메라 화면에도 같은 표가 HTML로 겹쳐 보이고(`live-board`), 항상 보임. 캔버스(사진)와 라이브 화면 표 크기는 항상 같은 비율(카메라 박스 실제 높이 × 0.06075)로 동기화되도록 `syncLiveBoardSize()`로 고정(높이/너비를 여러 차례 미세 조정해 현재 값에 도달: 너비 W×0.36, 줄높이 H×0.06075).
- 단지명/공종/위치는 **드롭다운(▾)** 방식, **하자내용은 드롭다운▾ + 마이크🎤 둘 다** 지원(목록에서 고르거나 말로 입력 가능). 물량도 마이크🎤 지원(파싱 없이 말한 텍스트 그대로 입력). 접수번호/동·호는 기존대로 마이크만.
- 음성 입력은 "항목 선택 후 그 항목만 말하기" 방식. 동/호는 "303동 704호" 또는 "303 704"(숫자만) 둘 다 인식. 하자내용/물량은 별도 파싱 없이 말한 내용 그대로 입력칸에 채움.
- 접수번호 형식: `6자리-2자리-5자리` 전부 사용자가 직접 입력. 예: `260121-00-12345`.
- 카메라 전환(전/후면) 버튼 제거 — 후면 카메라(`facingMode:'environment'`)만 고정 사용.
- **`CAM_RATIO`를 2.174 → 1.5로 변경 완료**(라이브 CSS `aspect-ratio`와 JS 상수 동시 수정) — 가로 폭은 유지하고 세로(위아래) 범위가 넓어져 화면이 덜 좁게 보임. 화면 비율 = 실제 촬영 비율이라 사진도 동일하게 적용됨.
- **촬영 전 필수 입력 검사 + 확인 팝업**: 셔터(📷) 클릭 시 `checkAndShoot()`가 항상 확인 팝업(`#confirm-modal`)을 띄움. 빈 칸이 있으면 팝업 안에 경고 문구가 뜨고 "촬영" 버튼이 비활성화됨. 팝업 안 버튼은 **"돌아가기 / 예외 / 촬영"** 3개 — "예외"는 빈 칸 여부와 무관하게 항상 눌러서 바로 촬영 가능(카메라 화면 밖의 별도 ⚡ 버튼은 오작동 방지를 위해 제거하고 팝업 안으로 옮김), "촬영"은 빈 칸 없을 때만 동작.
- 촬영 컨트롤은 셔터 버튼 1개만 남음(예외 버튼은 확인 팝업 안으로 이동). 촬영 후 마지막 사진을 작은 썸네일로 보여주고 탭하면 라이트박스로 크게 봄.
- **사진을 폰 갤러리(다운로드 폴더)에도 항상 저장**: 드라이브 업로드 성공/실패와 무관하게 촬영하면 `saveToGallery()`가 항상 실행됨(현장에서 인터넷이 안 되더라도 사진은 폰에 남도록). 갤러리 저장용 파일명만 별도로 **단지명을 맨 앞**에 두도록 `buildGalleryFname()`을 분리(드라이브 업로드용 `buildFname()`은 기존 규칙 그대로 유지) — 다운로드 폴더 안에서 단지명으로 정렬/검색하면 묶어서 찾기 쉬움. (진짜 폴더 분리는 웹앱 구조상 불가능하다고 안내함, 네이티브 앱 전환은 하지 않기로 합의)
- **작업목록을 묶는 기준을 "단지명+동+호+공종" → "접수번호"로 변경**: 같은 접수번호면 항상 같은 카드. 같은 단계(전/중/후)로 여러 장 찍으면 이제 마지막 1장만 덮어쓰지 않고 **배열로 전부 기록**됨.
- **사진 원본을 IndexedDB에 저장**: localStorage는 용량이 작아 이미지를 못 담으므로, 촬영한 사진(보드판 합성 포함)을 브라우저 내장 DB인 IndexedDB(`fb4_photos`)에 저장 → 폰을 끄고 다음날 다시 봐도 사진이 남아있음.
- **작업 카드에서 완료된 단계(✓ 도트)를 누르면 스와이프 뷰어**(`#shot-viewer`)가 열려서 그 단계에서 찍은 사진들을 좌우로 넘겨볼 수 있음(터치 스와이프 + 화살표 버튼). 도트 아래 라벨은 "N장"만 표시(시간은 제거함).
- **작업목록 표시 규칙**: 전/중/후 모두 끝난 작업은 목록 화면에서 제외(데이터는 보존, 화면만 숨김). 상단 건수도 화면에 보이는(미완료) 건수 기준.
- **작업목록 갱신 시점**: "구글 드라이브에 저장" 버튼이 아니라 **셔터 촬영 성공 시점**에 바로 반영됨(이전엔 업로드 버튼 눌러야 카드가 생겼음).
- 예전 데이터(단지명 기준 키, 단계당 1장 객체)는 `migrateJobs()`가 앱 시작 시 자동으로 새 구조(접수번호 키, 배열)로 변환함. 예전 사진은 이미지 자체가 없어서 뷰어에서는 빈 화면(시간/파일명 정보만)으로 보임.
- GitHub Pages(`https://seob-codes.github.io/site-photo-board/fieldboard-v4.html`)에 수동 업로드로 배포 중. 로컬 git 저장소 없음, push는 AI가 한 적 없음.
- 큰 구조 변경(접수번호 그룹핑+IndexedDB+스와이프 뷰어) 전에 `fieldboard-v4-backup-20260619.html`로 백업해둠 — 문제 생기면 이 파일 내용을 fieldboard-v4.html에 덮어써서 되돌릴 수 있음.

## 6. 하지 말 것 (제약)

- 단일 HTML 파일 구조 유지 — 파일 분리 금지(단, PWA 필수 파일인 `manifest.json`/`sw.js`/아이콘은 브라우저가 그 경로로 직접 요청하는 구조라 예외로 분리되어 있음 — 앱 본체 로직은 여전히 `fieldboard-v4.html` 하나에 다 있음)
- 새 외부 라이브러리 추가 금지 (추가 필요 시 먼저 물어볼 것) — IndexedDB, navigator.share, 서비스워커, 서명 패드는 모두 브라우저 내장 기능만 써서 구현, 외부 라이브러리 추가 없음
- 파일명 생성 규칙(`접수번호_단지명_동_호_공종_위치_하자내용_구분(순번).jpg`) 임의 변경 금지 — 변경 시 먼저 물어볼 것
- 라이브 화면 표(`live-board`)와 캔버스 표(`compositeOverlay`)의 크기/비율 계산식은 항상 같이 맞춰서 수정 — 한쪽만 고치면 촬영 전후 크기가 달라지는 버그가 재발함
- `CAM_RATIO` 상수는 라이브 CSS와 촬영 크롭 로직이 반드시 같은 값을 참조해야 함 — 한쪽만 바꾸면 화면/사진 비율이 어긋남
- git push/배포는 사용자가 직접 수행 — AI가 임의로 GitHub에 올리지 않음

## 7. 기술 스택

- HTML/CSS/Vanilla JS (프레임워크 없음)
- Web APIs: getUserMedia(카메라), SpeechRecognition(항목별 음성인식 + 산출식 음성계산), Canvas(사진 표 오버레이 합성 + 완료확인서 양식 그리기), localStorage(설정/작업목록/목록관리/기기코드 저장), **IndexedDB(촬영 사진·확인서 원본 이미지 저장, `fb4_photos`)**, `<a download>`(갤러리 저장 트리거), **`navigator.share`(폰 공유창으로 사진 내보내기)**, 마우스/터치 이벤트 기반 서명 패드(자체 구현, 라이브러리 없음), `Function()` 기반 안전한 수식 계산(산출식)
- 백엔드: Google Apps Script `doPost` 웹앱 → Google Drive 업로드 + 기기 코드 허가 확인(`checkAccess`)

## 8. 구현된 주요 기능

- 후면 카메라 촬영 (전면 전환 기능 없음, 단순화)
- 촬영 시 사진 좌하단에 흰 배경 표(단지명/동-호/위치/하자내용/작업사항[전중후점검]/물량) + 조직명 워터마크 합성, 라이브 화면에도 동일 비율로 미리 보임
- 촬영 직후 마지막 사진 썸네일 + 탭하면 크게 보는 라이트박스
- 항목별 마이크(접수번호/동·호/하자내용/물량) — 해당 항목 칸을 누르고 그 내용만 말하면 그 칸에 바로 입력됨
- 단지명/공종/위치 — 드롭다운(▾), 하자내용은 드롭다운▾+마이크🎤 둘 다, 목록은 설정 탭에서 추가/삭제 관리
- 전/중/후/점검 구분 버튼, 저장 후 다음 단계로 자동 전환 옵션
- **같은 접수번호**별로 작업목록 탭에 자동 그룹핑, 진행 상태(전/중/후) 시각화, 전/중/후 모두 끝난 작업은 목록에서 숨김
- 같은 단계로 여러 장 찍으면 전부 기록되어, 완료된 단계(✓ 도트)를 누르면 스와이프로 넘겨보는 사진 뷰어 제공
- 촬영 성공 시점에 바로 작업목록 반영 + 사진을 폰 갤러리(다운로드 폴더)와 IndexedDB에 항상 저장(드라이브 업로드 결과와 무관)
- 셔터 클릭 시 필수 항목 검사 + 파일명 확인 팝업("돌아가기/예외/촬영" 3버튼), 빈 칸 있으면 경고 후 "촬영" 비활성화, "예외"로 검사 우회 가능
- 파일명 규칙에 따라 자동 생성된 파일명으로 Google Drive 업로드 (실패해도 작업목록·갤러리 저장은 이미 됐으므로 오프라인에도 안전)
- 설정 탭: Drive 연동(Apps Script URL/폴더ID), **기기 코드**(승인 요청용, 복사 가능), 워터마크 설정(조직명), **완료확인서 정보(공사명/보수업체/작업자/현장대리인)**, 목록 관리(단지명/공종/위치/하자내용), 워터마크/자동업로드/자동진행 옵션 토글, 맨 아래 라이선스/개발자 안내 박스
- **완료확인서**: 촬영 화면 "📄" 버튼 → 미리보기 확인창 → 입주자/관리소 서명(이름+그림 서명, "(인)" 위에 겹쳐 합성) → 저장 시 갤러리+IndexedDB+작업목록 반영 + (자동업로드 토글 켜져있으면) 드라이브 업로드까지. 파일명 `접수번호_단지명_동_호_완료확인서_001`
- 작업목록에서 카드별 **🗑️ 삭제**(확인 팝업 거침) / **📤 내보내기**(사진+확인서를 폰 공유창으로 전달), 상단 "진행중/완료" 필터로 완료된 작업도 모아보기 가능
- **산출식 자동계산**: 산출식 입력(타이핑/마이크)하면 물량 칸에 결과 자동 채움, 보드판엔 "산출식=물량" 형태로 표시
- **무단 사용 방지**: 기기별 고유 코드 + 개발자 허가목록 기반 잠금 기능(현재 `LICENSE_CHECK_URL` 미설정으로 비활성 상태)

## 9. 핵심 규칙 / 데이터 형식

```
접수번호_단지명_동_호_공종_위치_하자내용_구분(순번).jpg
예: 260121-00-12345_문산당동3_303동_704호_도배장판_거실,주방_도배공사_전(1).jpg
```

- 접수번호는 `6자리-2자리-5자리` 전부 직접 입력 (고정 리터럴 없음).
- 순번(괄호 안 숫자)은 "물량"이 아니라 같은 작업+같은 구분으로 재촬영할 때마다 1씩 증가하는 순번임.
- "물량"은 입력 폼에 신규 추가 완료. 표(보드판)에는 표시되지만 **파일명 생성(`buildFname`/`getFieldMeta`)에는 포함되지 않음**.
- 사진 위 표에는 단지명/동-호/위치/하자내용/작업사항(전중후점검)/물량 6줄이 보임. 접수번호·공종은 파일명에만 쓰이고 표에는 안 나타남(하자내용은 파일명+표 둘 다 사용).
- 촬영 시 필수 항목(접수번호/단지명/동/호/공종/위치/하자내용/촬영구분 — 물량 제외) 중 빈 칸이 있으면 파일명 확인 팝업에 경고가 뜨고 "촬영" 버튼이 막힘. 팝업 안 "예외" 버튼으로 이 검사를 건너뛸 수 있음.
- 갤러리 저장용 파일명(`buildGalleryFname()`)은 **단지명이 맨 앞**이고, 드라이브 업로드용 파일명(`buildFname()`)은 기존 규칙(접수번호가 맨 앞) 그대로 — 둘은 서로 다른 파일명이지만 같은 사진.
- 작업목록은 **접수번호**가 같으면 같은 카드로 묶임(단지명+동+호+공종 기준에서 변경됨).
- 접수번호/동/호는 **숫자=기존 자동 포맷, 한글 섞이면 예외 텍스트로 그대로 저장**되는 이중 모드(현장에서 "선조치", "관리동 기타" 같은 비정형 기록이 필요할 때 사용). 자동 삽입된 대시(-)는 예외 판단에서 제외됨(버그 수정 완료).
- 작업 카드 제목(큰 글씨)은 **하자내용** 기준(예전엔 공종 기준이었음).
- 산출식은 숫자/사칙연산(`+-*/`)/괄호/공백만 허용되고, 그 외 글자가 섞이면 계산하지 않음(안전장치). 산출식이 비면 물량도 같이 비워짐.

| 카카오톡으로 "📤 내보내기" 했을 때 사진 여러 장이 한 메시지로 안 묶이고 한 장씩 따로따로 전송됨 | 우리 앱은 `navigator.share()`로 사진들을 폰 공유 기능에 넘기기까지만 하고, 그 이후 "묶을지/한 장씩 보낼지"는 받는 앱(카카오톡)이 결정 — 코드로 제어 불가능한 영역 | 알려진 한계로 안내함. zip으로 묶으면 받는 사람이 압축을 풀어야 해서 더 불편해짐. 다른 앱(문자 등)에서는 다르게 동작할 수도 있음, 필요시 추가 테스트 |

| **(중대) 앱 전체 버튼/탭이 갑자기 아무것도 클릭 안 됨** | 마커 드래그 종료 함수(`_markerDragEnd`)가 드래그 중이 아닐 때도 모든 mouseup/touchend에 `preventDefault()`를 호출해서, 터치 후 발생해야 할 클릭 이벤트 자체가 전부 차단됨 | `if(!_markerDrag.active) return;` 조건 추가해서 드래그 중일 때만 막도록 수정. **앞으로 전역(document) 리스너 추가 시 "조건 없이 항상 실행되는 부분"이 있는지 반드시 직접 재확인** — 문법 검사만으론 이런 로직 버그를 못 잡음 |
| **(중대) 기기 코드 승인 검사가 항상 통과됨**(시트를 비워도 잠금이 안 걸림) | `fetch`에 `Content-Type: 'application/json'` 헤더를 쓰면 브라우저가 CORS 사전검사(OPTIONS)를 먼저 보내는데, Apps Script는 OPTIONS를 처리하는 코드가 없어서 405로 거부함. 이 오류가 fail-open(네트워크 오류 시 통과) 로직에 걸려 조용히 통과되어 버그를 못 느꼈음 | `Content-Type: 'text/plain;charset=utf-8'`로 변경해서 사전검사 자체가 안 생기게 해결. Apps Script는 헤더와 무관하게 본문을 그대로 읽으므로 서버 코드는 그대로 둬도 됨. **사진/완료확인서 드라이브 업로드도 같은 패턴이라 동일 수정 — Apps Script로 fetch 보낼 땐 항상 text/plain을 쓸 것** |
| 완료확인서 안내 그림(SVG)의 좌표 숫자가 실제 코드와 다름(50px 오차) | 좌표를 손으로 계산하다가 중간 단계(710+80 대신 660+80)를 잘못 더함 — 추측 계산은 이런 실수가 잘 안 보임 | 이후로는 좌표를 바꿀 때마다 `node -e`로 직접 코드 흐름을 시뮬레이션 실행해서 값을 뽑고, 그 값으로 안내 그림을 다시 그리는 절차로 변경. **좌표/계산 관련 변경은 추측하지 말고 항상 실행해서 확인할 것** |

## 13. 결정 사항 & 이유 (왜 이렇게 짰는지)

- 사진 오버레이를 표(테이블) 디자인으로 변경 → 사용자가 실제 현장에서 쓰는 종이 보드판 양식 사진을 레퍼런스로 제공, 그대로 재현
- 라이브 화면에도 같은 표를 항상 보이게 함 → 사용자가 "촬영 전엔 안 보이다가 구분 눌러야 보이는 게 불편하다" → 항상 보이게, 입력하는 대로 채워지게 변경
- 단지명/공종/위치는 드롭다운+목록관리로, 마이크는 접수번호/동호에만 → 사용자가 "목록표가 있는데 마이크까지 있을 필요는 없다"고 판단
- 음성 입력을 "전체 한 번에 말하기"에서 "항목별로 선택해서 말하기"로 전환 → 사용자가 순서 상관없이 원하는 항목만 정확히 입력하고 싶다고 요청, 한 문장 파싱 방식보다 정확도가 높음
- 카메라 전환(전/후면) 버튼 삭제 → 사용자가 "사진만 찍으면 된다"며 불필요한 기능으로 판단, 후면 고정
- `CAM_RATIO` 상수를 라이브 CSS와 촬영 크롭이 공유하도록 통일 → 화면에서 보던 범위와 실제 찍히는 범위가 달랐던 문제(폰을 세로로 들면 사진이 다르게 찍힘) 해결, 이후 표 크기 동기화 버그도 같은 패턴으로 해결
- 표/마이크/드롭다운 버튼 디자인을 입력칸과 같은 줄에 배치 → 처음엔 라벨 줄에 뒀다가 사용자가 "입력칸 옆으로" 요청, 순서도 input→▾→🎤로 맞춤
- `CAM_RATIO`를 2.174→1.5로 변경 → 사용자가 "화면 박스 높이가 너무 좁다"고 요청, 가로는 유지하고 세로 범위 확대(화면=촬영 비율 항상 동일하게 유지하는 기존 규칙 그대로 적용)
- 하자내용에 마이크를 추가(드롭다운과 병행) → 목록에 없는 새 하자도 말로 빠르게 입력하고 싶다는 요청
- 물량을 "비고" 칸 자리에서 별도 입력 항목으로 분리 + 파일명 제외 → 표에는 보여야 하지만 파일명 규칙(접수번호_단지명_동_호_공종_위치_하자내용_구분(순번))은 기존 그대로 유지하고 싶다는 요청
- 보드판 라벨 배경을 회색→흰색으로 통일 → 사용자가 깔끔한 흰색 톤을 선호
- 촬영 전 필수 항목 검사 + 파일명 확인 팝업 + "예외" 버튼 도입 → 사용자가 "빈 칸으로 잘못 찍히는 걸 막고 싶다"면서도 "정보 없이 찍어야 하는 경우도 있다"고 요청 → 기본은 엄격히 막되, 예외 버튼으로 빠르게 우회할 수 있는 절충안으로 설계
- 사진을 갤러리에도 항상 저장(드라이브 결과와 무관) → 사용자가 "현장을 벗어나면 다시 찍을 수 없다"며 오프라인/업로드실패 대비 요청. 처음엔 "업로드 성공시에만"(A안)으로 정했다가 이 이유로 "항상"(B안)으로 번복
- 갤러리 저장 파일명만 단지명을 맨 앞으로 분리 → "네이티브 앱처럼 폴더별 저장은 불가능하다"고 안내 후, 절충안으로 파일명 정렬을 통한 구분을 사용자가 수락
- 작업목록 묶는 기준을 단지명+동+호+공종 → 접수번호로 변경 → 사용자가 "같은 접수번호인데 카드가 2개로 생기면 안 된다"고 요청(같은 접수번호=하나의 일이라는 현장 인식 반영)
- 같은 단계 여러 장을 배열로 저장 + IndexedDB + 스와이프 뷰어 도입 → 사용자가 "전 사진을 10장 찍었을 때 그걸 다음날 다시 보고 중 사진을 어떻게 찍을지 참고하고 싶다"고 요청. localStorage 용량 한계 때문에 이미지 저장은 IndexedDB로 분리
- 작업목록 갱신 시점을 "드라이브 업로드 버튼 클릭" → "셔터 촬영 성공 시점"으로 앞당김 → 사용자가 "촬영이 성공하면 바로 목록에 보여야 한다"고 요청
- 전/중/후 모두 끝난 작업을 목록에서 숨김(데이터는 보존) → 사용자가 완료된 작업까지 계속 보이는 게 번잡하다고 판단, 다만 완전 삭제는 위험하다고 판단해 화면 숨김만 적용
- "예외" 버튼을 카메라 화면 밖에서 확인 팝업 안으로 이동 → 사용자가 "셔터 옆에 버튼이 2개라 실수로 예외를 누르는 경우가 많을 것 같다"고 우려, 오작동 방지를 위해 팝업 안쪽(빈 칸 있을 때만 의미 있는 위치)으로 옮김
- 앱 이름을 "써비의 현장보드판"으로 변경 → 사용자가 "누가 만들었는지는 알아야지"라며 요청
- 공종 키워드 자동 아이콘 / 도트 연결선 삭제 → 사용자가 각각 "필요 없어 보인다"고 판단, AI도 기능상 손실 없다고 동의
- 완료확인서 기능을 **이미지(JPG)**로 만들기로 결정(진짜 .xlsx 생성 아님) → "사진처럼 폰으로 찍어서 제출하는 용도면 충분하다"는 사용자 확인. 진짜 엑셀 파일을 만들려면 새 외부 라이브러리(SheetJS 등)가 필요해서 지금 구조(단일 HTML, 라이브러리 추가 금지)와 안 맞음
- 완료확인서의 공사명/보수업체/작업자/현장대리인을 설정 탭에 저장 → "매번 입력하지 않고 한 번 설정해두면 재사용"하고 싶다는 요청, 단 영구 고정이 아니라 언제든 수정 가능하게 설계
- 완료확인서도 촬영처럼 "바로 저장"이 아니라 "미리보기 확인창"을 거치게 함 → 사용자가 "내용이 맞는지 확인하고 싶다"고 요청, 촬영 확인창과 같은 패턴 재사용
- 서명을 캡처해 이미지로 합성(별도 디지털 서명 서비스 연동 안 함) → 단일 HTML/무료 구조 유지가 우선이라 마우스/터치로 그리는 자체 구현 서명 패드로 충분하다고 판단
- 서명 이미지를 "(인)" 글자 옆이 아니라 **겹치게** 배치 → 사용자가 실제 양식에서 도장이 "(인)" 위에 찍히는 모양을 기대한다고 피드백
- 작업목록 "확인서" 도트=보기, 버튼=추가생성으로 역할 분리 → 사용자가 "도트랑 버튼이 중복되는 것 같다"고 지적, 전/중/후 패턴(도트=보기, 버튼=재촬영)과 일치시켜 해결
- 작업 삭제 확인을 브라우저 기본 confirm()→커스텀 모달로 교체 → 사용자가 "너무 조잡하게 뜬다"고 피드백, 촬영 확인창과 같은 디자인으로 통일
- "📤 내보내기"를 완전 자동 문자전송이 아니라 `navigator.share` 공유창 방식으로 구현 → 브라우저가 SMS를 직접 보낼 권한이 없어서 기술적으로 완전 자동은 불가능, "공유 후 마지막 전송만 사용자가" 하는 절충안으로 결정. 사용자가 "일단 만들고 불편하면 삭제하면 된다"고 동의해 가볍게 시도
- PWA 전환 진행 → 기능 다듬기를 충분히 마친 뒤 사용자가 먼저 "PWA부터 만들어두 괜찮아?"라고 요청. 서비스워커를 네트워크 우선 방식으로 설계 → 예전에 "PWA로 바꾸면 캐시 때문에 업데이트가 늦게 반영될 수 있다"고 미리 안내했던 우려를 해결하기 위한 선택
- manifest.json/sw.js/아이콘 파일을 분리 생성 → PWA 특성상 브라우저가 이 파일들을 직접 요청하기 때문에 단일 HTML 안에 합칠 수 없음(기술적 제약), 다만 앱 본체 로직은 fieldboard-v4.html 하나에 그대로 유지해 "단일 HTML 유지" 원칙의 취지는 지킴
- 접수번호/동·호에 "한글 예외" 모드 추가 → 사용자가 "접수번호만으로 선조치/관리소요청/소장님요청 같은 걸 적을 때가 있다"고 요청. 기존 6-2-5 자동 포맷은 그대로 유지하되, 숫자가 아닌 글자가 섞이면 포맷을 건너뛰는 방식으로 절충(코드 하나로 두 모드 자동 구분)
- 접수번호 한글 예외 입력 시 띄어쓰기 제거하고 붙여쓰기로 저장 → 사용자가 명시적으로 요청
- 동/호 마이크에 한글 인식 추가 → 사용자가 "관리동기타" 같은 예시를 듦, 기존 숫자 파싱 로직이 실패하면 단어 단위(앞=동, 뒤=호)로 나누는 fallback을 추가해 해결
- "파일명 설정" 카드 → "워터마크 설정"으로 이름 변경 → 사용자가 실제 기능(워터마크 표시용)과 이름이 안 맞는다고 질문, 정확하게 수정
- 설정 탭 placeholder(예시 텍스트) 전부 제거 → 사용자가 명시적으로 요청, 빈 칸으로 시작하게 변경
- 작업 카드 제목을 공종→하자내용으로 변경 → 사용자가 카드 위쪽 큰 글씨에 무엇이 표시되는지 묻고 하자내용으로 바꿔달라고 요청
- 산출식 칸을 신설하고 결과가 물량에 자동으로 들어가게 함 → 사용자가 "산출식도 필요하다"며 `(2*2)*2.3-(1*1)` 예시를 듦. 안전한 계산을 위해 숫자/사칙연산/괄호만 허용하는 정규식 검사 후 `Function()`으로 계산
- 산출식에도 마이크 추가(곱하기/나누기/더하기/빼기 인식) → 사용자가 추가 요청
- 보드판 표시를 "물량"에서 "산출식/물량"(산출식=물량 형태)으로 변경 → 사용자가 명시적으로 요청, 산출식 과정과 결과를 같이 남기고 싶다는 의도
- 작업목록 완료 작업 "무조건 숨김" → "진행중/완료 필터"로 전환 → 사용자가 직접 "후" 사진 찍고 카드가 사라지는 걸 보고 당황, 의도된 기능이었지만 막상 체감해보니 불편하다고 판단해 필터 방식으로 개선 요청
- 무단 배포/사용 방지를 위한 기기 코드 승인 기능 도입 → 사용자가 지인에게 공유한 앱이 또 다른 사람에게 퍼지는 걸 걱정, "불편하더라도 보안이 1번"이라고 판단. 완전 차단은 기술적으로 불가능하지만, 이미 보유한 Apps Script(드라이브 서버)를 활용해 개발자가 통제 가능한 허가목록 방식을 채택 — 네트워크 오류 시엔 fail-open(통과)으로 현장 운영 안정성 우선
- 라이선스 경고 박스에서 전화번호 제외 → 사용자가 참고 이미지를 보내며 전화번호 포함을 요청했으나, AI가 "공개 배포 코드에 전화번호가 노출되는 보안 위험"을 안내하자 사용자가 직접 위험성을 인지하고 제외하기로 결정
- 완료확인서 접수내용/보수내용 가운데정렬 → 취소, 왼쪽 정렬로 복구 → 한 번 적용해봤다가 사용자가 기존 느낌이 더 낫다고 판단해 되돌림
- 보수내용 항목을 공종-위치-하자내용 → 위치-하자내용-물량으로 변경 → 사용자가 "보수내용에 실제로 들어가야 할 정보가 다르다"고 지적
- 현장대리인 도장을 "(인)" 텍스트와 함께(겹쳐서) 표시 + 세로형(정사각형 아님) 테두리로 변경 → 처음엔 "(인)" 텍스트 없이 도장만 그렸는데, 사용자가 입주자/관리소 서명처럼 "(인)" 위에 겹치는 느낌과 세로로 긴 모양을 요청
- 확인자(입주자)/(관리소) 줄을 오른쪽으로 이동 → 사용자가 명시적으로 요청
- 마커 기능을 "아무데나 탭하면 찍힘" → "모드 버튼을 켜야만 동작" 방식으로 변경 → 사용자가 실수로 화면 탭했을 때도 마커가 찍히는 부작용을 우려, 명확한 on/off 스위치 도입
- 마커 크기조절을 ＋/－ 버튼 → 마커에 달린 드래그 손잡이 방식으로 변경 → 사용자가 "더 직관적인 방식"과 "가로/세로 따로(타원, 늘어난 화살표)" 조절을 원해서, size(숫자1개) 구조를 width/height(가로세로 따로) 구조로 재설계
- 완료확인서 마커/디자인 관련 기능들은 모두 "확정 전 연습 단계"로 명시 → 사용자가 실제 현장 사진으로 테스트 후 최종 디자인을 정하고 싶다고 요청, 가볍게 시도하고 바꿀 수 있게 설계
- 전역 클릭 차단 버그 발견 후 작업 방식 변경 → 사용자가 "코드 수정 후 실수 부분 재확인하기로 했었잖아"라고 직접 지적. 이후로는 문법 검사만으로 "완료"라고 보고하지 않고, 전역 이벤트 리스너 등 사이드 이펙트 가능성을 직접 다시 읽고 확인하는 절차를 추가하기로 함
- 기기 코드 허가목록을 Apps Script 코드 배열(`ALLOWED_DEVICES`) → 구글 시트 관리로 변경 → 사용자가 "구글 시트에 새 시트 만들어서 거기서 관리하고 싶다"고 직접 요청. 다른 본인 프로그램들도 같은 패턴(`OOO 허가목록` 시트)으로 관리하고 있어서 일관성 있게 적용, 코드 재배포 없이 시트 수정만으로 승인 가능해짐
- `LICENSE_CHECK_URL`을 별도 상수로 안 두고 기존 드라이브 연동 주소(`#s-url`)를 재사용 → 같은 Apps Script 프로젝트가 사진 업로드와 기기 승인 검사를 모두 처리하고 있어서, 별도 URL을 또 입력받게 하면 사용자에게 불필요한 입력 단계가 늘어남
- 좌표 변경 시 안내 그림(SVG)을 추측으로 다시 그리지 않고 매번 node로 실행 검증 → 안내 그림 숫자가 50px씩 틀렸던 걸 사용자가 실제 앱 스크린샷과 비교해서 직접 발견, 손 계산은 중간 단계를 빠뜨리기 쉽다는 교훈
- GitHub 저장소를 Public으로 유지(Private 전환 보류) → 무료 계정 기준 Private 저장소는 GitHub Pages 배포가 막히거나 제한될 수 있어, 배포된 웹앱이 갑자기 안 열리는 위험을 사용자가 직접 판단해 보류 결정
- 무단 사용 방지는 "링크가 그냥 퍼지는 것"을 막는 선까지만 — 클라이언트 전용 웹앱 구조상 소스 코드를 통째로 복사해 검사 로직을 지우거나 네트워크 요청을 차단하면 우회 가능함을 사용자에게 설명, 완전한 차단은 기술적으로 불가능하다는 현실적 한계에 합의(기존 "허가목록 방식" 결정과 같은 맥락)
